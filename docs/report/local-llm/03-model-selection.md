## 1. 結論

**スループット重視: Gemma 4 E2B Q4_K_S ／ 品質重視: Gemma 4 E4B Q4_K_M。Q5_K_M は backup、Q8_0 は除外。**

| 項目 | 内容 |
|---|---|
| **スループット重視の第一候補** | **Gemma 4 E2B Q4_K_S**（約 3 GB） — E4B Q4 の約 2 倍のスループット、品質は 93% |
| **品質重視の第一候補** | **Gemma 4 E4B Q4_K_M**（約 5 GB） — 量子化 sweep 内での品質トップ |
| **backup** | **Gemma 4 E4B Q5_K_M**（約 5.5 GB） — Q4 と量子化階層が違う |
| **除外** | Q8_0（約 8 GB） — 最遅 AND 品質もトップではない、Q4 に完全に支配される |
| **判定 host** | LM Studio（Phase 3 勝者）に固定 |

### なぜ候補が 2 つに分岐するか

2026-04-20 に Google 公式の **Gemma 4 E2B**（`google/gemma-4-E2B-it`、unsloth GGUF 版）を測定に加えたところ、
E4B 量子化 sweep とは別次元の速度・サイズの優位が出た。そのため「量子化レベルの選択」ではなく
「モデルバリアント × 量子化の選択」として結論を再構成している。

### Gemma 4 E2B Q4_K_S（スループット重視）の決め手

1. **最速**（S2 26.0 秒 vs E4B Q4 46.8 秒（-44%）、S3 33.4 秒/画像 vs 81.9 秒（-59%））— tok/s は 27.8 で E4B Q4 の約 1.9 倍
2. **メモリ最小**（ディスク 2.97 GB / 推論時 RAM 約 4.5 GB）— target mini PC（32 GB RAM）で圧倒的に余裕
3. **抽出品質は E4B Q4 の 93%**（0.709 / 0.760）— backup 閾値 80% を上回り、tc02 では E4B Q4 を上回る（0.750 vs 0.650）
4. **判断品質は E4B 全量子化と統計的に同等**（n_runs=3 で Q8 0.661 > Q4 0.619 > E2B 0.582 > Q5 0.579、全差 1σ 以内）— 単発ランで首位に見えた優位はノイズで消えるが、「明確に劣っている」わけでもない
5. **4/4 完走**（S3 timeout なし）

> ⚠ 注: E2B は **判断テストの std が全候補中最大（±0.144）**。「良い日は E4B 級、悪い日は明確に落ちる」性質のため、業務投入時は **判断系出力は n=2〜3 で self-consistency を取る** か、**E4B にフォールバック**する設計が安全。

### E4B Q4_K_M（品質重視）を残す理由

- E4B 量子化 sweep 内で **品質トップ**（0.760 ≧ Q5 0.722 ≧ Q8 0.716）
- 特に **tc03（複雑アーキ図）は E4B Q4 が 0.769、E2B は 0.577** — 複雑な構造解釈では E4B が優位
- tc04（テキスト文書 OCR）は E4B Q4 が 0.955、E2B が 0.864 — 純 OCR でも差が出る
- 速度的制約が厳しくない場合は E4B Q4 の方が安心

### backup を Q5_K_M とする理由

- 品質は E4B Q4 の **95%**（0.722 / 0.760）— backup 閾値 80% を大きく上回る
- 量子化階層が違う ため、将来 Q4 のモデル更新で問題が出ても Q5 が同時に壊れる可能性は低い
- 速度も実用域内（S2 58.5 秒）、ロード方法も同じで切替コストが低い

### Q8_0 を除外する理由

- 速度は最遅（S2 77.4 秒、Q4 比 +65%）
- 品質も Q5 と同等（0.716 vs 0.722）で、より重い割に得るものがない
- Q5 と比べても backup として優先する理由がない

## 4. 結果から読み取れること

### 4.1 速度はファイルサイズに比例（E4B 内）／ E2B は別軸で支配的

CPU-only 推論ではメモリ帯域がボトルネック → 量子化サイズが直接スループットに反映：

- Q5_K_M は Q4_K_M より約 25% 遅い（10% 大きいファイル + 量子化デコード分のオーバーヘッド）
- Q8_0 は Q4_K_M より約 65% 遅い。S2 で 9.5 tok/s は「CPU 使用可能」水準のぎりぎり上
- **E2B Q4_K_S（2.97 GB）は E4B Q4_K_M（4.98 GB）より約 1.9 倍速い** — パラメータ数（4.6 B vs 7.5 B）が支配的
- E2B と E4B の速度差（約 2 倍）は E4B 内の量子化差（Q4→Q8 で 1.65 倍）より大きい

### 4.2 品質は E4B 3 量子化では区別不能、E2B は 93% レベル（抽出テスト）

「量子化が軽ければ軽いほど品質が低い」という直感に反して、E4B 内では Q4 が平均トップ。これは「N=4 のテストセットでは差は judge 変動の範囲内」と読むのが妥当：

- E4B 3 量子化の avg 差は 0.04 — judge（Gemini 2.5 Flash）自体の sampling variance と同等
- tc02 が E4B 全量子化で最難（0.450〜0.650）— UI before/after 変化の列挙は苦手
- tc04 が E4B 全量子化で最易（0.955〜1.000）— 純テキスト OCR は E4B の native 多言語 OCR の得意領域

E2B は E4B Q4 の **93%**（0.709 / 0.760、抽出テスト平均）— judge 変動を超える有意差は 1 ケースのみ：

- **tc02 で逆転**（E2B 0.750 > E4B Q4 0.650）— E2B の方が UI 変化列挙が得意
- **tc03 で差（-0.192）**（E2B 0.577 vs E4B Q4 0.769）— 複雑アーキ図の要素数・階層関係では E4B が優位
- tc01 / tc04 は近似（差 ±0.091 以内）

つまり「E2B は E4B より弱い」ではなく、**「得意分野が違う」**。UI 変更検知や高速さが効くユースケースでは E2B、
複雑な構造図の網羅性が効くユースケースでは E4B Q4 を使い分ける構成が自然。

### 4.2.b 判断テスト（reasoning）の結論 ― 量子化差は統計的にノイズ内

n_runs=3 平均:

- **Q8 0.661 ±0.104**（名目首位）> Q4 0.619 ±0.082 > E2B 0.582 ±0.144 > Q5 0.579 ±0.063
- 全差が 1σ 以内。判断テストでは candidate 間に統計的差はない。

当初（n_runs=1）の「E2B が判断テストで全 E4B 量子化を上回る」という見立ては、E2B の tc02_judge が 1.000 を取った 1 回のランに依存していた artifact。3 回平均では 0.738 に収束し、Q8 (0.786) / Q4 (0.738) と並ぶ。

**量子化とサイズが判断力に与える効果（n=3 から読めるもの）:**

- tc02_judge（UI 変更の工数見積・リスク・並列性）は全候補とも 0.71〜0.79 に集中。reasoning point 数が 7 と少ないためスコア粒度が大きい。
- tc03_judge（AWS 構成の視覚読解）は全候補とも 0.43〜0.54 に留まる。これは Phase 4 全体の **共通弱点**であり、量子化やサイズでは解消しない。より強い VLM かタスク別プロンプトが必要。
- E2B は std が最大（±0.144）で、"run ごとに出来がブレやすい" という実務上の性質が明確になった。

**§1 への反映:** 「スループット重視＝E2B」の根拠は速度 + 抽出品質 93% に絞る。「E2B が判断で E4B より上」の主張は撤回。

### 4.3 選定ルールチェック

**E4B Q4_K_M（品質重視）:**

- ✅ 第一候補は品質の（タイ）トップ（0.760）
- ✅ 第一候補は S3 を完走（4/4 成功）
- ✅ 第一候補は最速（最速の 2 倍以内 → 自身が最速、※E2B 除く E4B 内で）
- ✅ Backup は第一候補品質の 80% 以上（Q5 は 95%）
- ✅ Backup は異なる量子化階層（Q4→Q5）で redundancy あり

**E2B Q4_K_S（スループット重視）:**

- ✅ 品質は E4B Q4 の 93%（backup 閾値 80% を大きく上回る）
- ✅ S3 完走（4/4 成功）
- ✅ 最速（全候補中 1 位、E4B Q4 の 1.9 倍）
- ✅ メモリ最小（ディスク 2.97 GB / 推論時 RAM 約 4.5 GB）
- ⚠️ 複雑な構造図（tc03 系）では E4B Q4 より -0.19 の差あり — 用途によっては E4B fallback を準備

## 5. 留保事項

- **N が小さい:** テストケース 4 件、fact 合計約 90 件。品質数値の sampling variance が大きい。10 ケース版にすれば比較がもっと締まる。
- **Judge は単一（ただし人手クロスチェック UI + マルチラン導入済み）:** LLM judge は Gemini 2.5 Flash のみだが、判断テストは `--n-runs 3` で 3 回独立に describe+judge を回しており、per-run 内訳は `{quant}_{tc}_run{N}_scores.json` に保存、per-fact の verdicts[] / agreement は `{quant}_{tc}_scores.json` に集約済み。抽出テストは現状 n_runs=1 だが、レビュアーが疑う項目があれば同じ `--n-runs N` オプションで追加計測できる（[§3 LLM judge に加えた人手クロスチェック](#llm-judge-に加えた人手クロスチェック) の UI も判断テスト対応済み）。人手 verdict と LLM verdict、あるいは LLM run 間の agreement が 67% 未満（3 回中 2 回以上バラついた）のケースは特に見直す価値が高い。
- **デフォルト sampling:** 全候補とも LM Studio のデフォルト `temperature` / `top_p` / `max_tokens` を使用。`temperature=0` の決定論的実行で 1 つのノイズ源を消せるが、絶対値は変わる可能性あり。
- **日本語プロンプトのみ:** `tc01..tc04` はいずれも日本語の質問。英語や混合言語のプロンプトでは未評価。
- **E2B の量子化は Q4_K_S のみ測定:** unsloth 配布の LM Studio デフォルト選択が Q4_K_S だったため、E2B 側で Q4_K_M / Q5_K_M の比較は未実施。E4B Q4_K_M 相当の量子化で追加計測すると E2B 内部の量子化感度も見える。

## 6. 次のアクション

1. **Phase 6（target mini PC 検証）では候補を 2 段構えで流す:**
   - 第 1 候補 **E4B Q4_K_M** を先に流し、2 倍以下のスローダウンを確認できれば品質重視で採用
