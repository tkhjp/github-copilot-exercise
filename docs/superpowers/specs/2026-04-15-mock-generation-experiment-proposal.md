# 調査・評価方針案

> **テーマ:** 仕様駆動開発における Web アプリケーションへの機能追加時の LLM によるモック生成能力評価

---

## 1. 背景

- 当組織では、仕様駆動開発（Specification-Driven Development）の文脈で、AI コーディングエージェント（GitHub Copilot、Codex 等）を活用した初期モック生成の実用性を検討している。
- 既存の Web アプリケーションに機能を追加する際、LLM に与えられる入力情報は利用者の役割・契約形態により異なる:
  - **外部委託先**: 画面キャプチャ（スクリーンショット）のみ閲覧可能
  - **委託先（中間階層）**: レンダリング済み HTML までアクセス可能
  - **内製開発チーム**: 完全なソースコードにアクセス可能
- 本調査は、**入力情報の粒度** と **仕様書の記述粒度** が LLM のモック生成品質に与える影響を定量評価する。

---

## 2. 調査目的

- 機能追加を想定したモック生成タスクにおいて、異なる入力条件・仕様粒度下での LLM の出力品質を比較する。
- Copilot / Codex の 2 種類のエージェントを同条件で比較し、製品特性の違いを明らかにする。
- 仕様駆動開発フロー上、**どの入力条件・仕様粒度であれば実用水準のモックが得られるか** を示し、現場適用の推奨条件を提示する。

---

## 3. 調査対象・スコープ

### 3.1 対象

| 区分 | 内容 |
| --- | --- |
| テスト対象アプリ | `demo-app`（React 19 + TypeScript + Tailwind CSS v4 + Vite）|
| エージェント | GitHub Copilot Coding Agent / OpenAI Codex Coding Agent（Partner Agent 経由）|
| 実行環境 | GitHub-native（Issue → Bot assignment → PR 生成フロー）|
| 評価対象 | 生成された PR の差分（コード）|

### 3.2 スコープ内

- 追加機能モックの生成品質評価（定量・定性）
- 入力条件別・仕様粒度別・エージェント別の比較
- 現場適用可能な推奨条件の整理

### 3.3 スコープ外

- 本番品質のプロダクションコード生成評価
- 大規模リファクタリングや既存ロジックの深い改変を伴うタスク
- 長期運用・保守性の評価

---

## 4. 前提・制約

- 本実験の「モック」は、**機能が視認・操作でき、`npm run dev` で描画確認できる水準** を指す。型安全性や lint clean、既存コードとの完全な統合は**要求しない**。
- Copilot / Codex はいずれも GitHub の Coding Agent フレームワーク上で動作する Partner Agent であり、IDE 内の Copilot Chat 体験とは**挙動が異なる可能性** がある。本報告の結論はこの前提下での比較である。
- サンプル数は各条件 1 回実行（計 48 run）とし、LLM の確率的挙動を完全には制御しない。代表的な傾向を把握する「軽量 PoC」として位置づける。

---

## 5. 評価対象タスク

`demo-app` 仕様書（`SPEC.md`）を踏まえ、2 つの機能追加タスクを設定する。

### タスク ①: ダッシュボード KPI 期間切替トグルの追加

- **概要**: Dashboard 画面の KPI カード群に「今月 / 今四半期 / 今年」の切替トグルを追加し、選択期間に応じて値を切り替える
- **特徴**: 既存コンポーネント内の局所改修、1〜2 ファイルの変更で完結
- **難度**: 軽（ローカルな UI 拡張）

### タスク ②: 通知センターページ（`/notifications`）の新設

- **概要**: AI エージェントからの通知を一覧するページを新設。Header の通知ベル・Sidebar からの導線を追加、既読/未読フィルタ機能付き
- **特徴**: 新規ページ追加＋複数既存ファイルへの配線、アーキテクチャ横断
- **難度**: 中（複数ファイル跨ぎ・新規画面）

---

## 6. 実験変数

### 6.1 入力条件（4 水準）

| ID | 入力 | 想定役割 |
| --- | --- | --- |
| A1 | スクリーンショット画像（PNG 直接添付） | 外部委託先（画像のみ閲覧可）|
| A2 | スクリーンショットを Gemini Vision で Markdown 化した構造化テキスト | 画像を事前テキスト化する運用フロー |
| B | Playwright で取得したレンダリング後の HTML | 中間階層（UI 構造参照可、ソース不可）|
| C | 関連 TypeScript ソースファイル抜粋 | 内製開発（完全情報）|

### 6.2 仕様粒度（3 水準）

| ID | 形式 | 概要 |
| --- | --- | --- |
| L1 | 一文指示 | 「KPI に期間切替トグルを追加してください」レベル |
| L2 | 自然言語詳細 | 目的・要件・非機能要件を箇条書きで記述 |
| L3 | 構造化仕様 | User Story + UI 要素表 + データ仕様表 + 受入条件（AC）checklist |

### 6.3 エージェント（2 水準）

- GitHub Copilot Coding Agent（`@copilot`）
- OpenAI Codex Coding Agent（`@codex`）

### 6.4 実行マトリクス

**4 入力 × 3 仕様粒度 × 2 エージェント × 2 タスク × 1 回 = 計 48 run**

---

## 7. 実行方法（GitHub-native Automation）

1. **準備フェーズ**:
   - `demo-app` を複製した実験用リポジトリを準備
   - 入力素材（スクリーンショット、HTML、ソース抜粋）を事前作成
   - 3 粒度 × 2 タスク = 6 種類の仕様書を事前作成
   - Playwright E2E テスト（タスクごと）を事前作成
2. **実行フェーズ**:
   - スクリプトが Matrix を順次展開し、各 cell について Issue を作成
   - 該当エージェントを Issue に assign（`@copilot` または `@codex`）
   - エージェントが自動で PR を生成
   - PR 生成までを poll（タイムアウト 30 分）
3. **評価フェーズ**:
   - 各 PR に対し `npm run build` / Playwright E2E / LLM-judge を自動実行
   - 結果を `runs/{cell_id}/` に保存

---

## 8. 評価指標

モック生成の品質を 5 次元で評価し、加重合計で総合スコアを算出する。

| # | 次元 | 測定方法 | 重み |
| --- | --- | --- | --- |
| D1 | 基本起動 | `npm run dev` でエラーなく描画できるか（build 失敗は fail、型 warning は許容）| 15% |
| D2 | 機能可見性 | Playwright で追加機能が画面上に存在し最低限の操作ができるか | 20% |
| D3 | 仕様一致性 | LLM-judge が L3 の受入条件（AC）checklist を present / partial / missing で判定 | 40% |
| D4 | コード適合度 | LLM-judge が命名規則・Tailwind トークン使用・既存ファイル構成踏襲を評価 | 25% |

> **注**: 次元番号は本方針書での通番。詳細設計では内部的な ID が追加される。

### 付帯分析: 出力形態の分類

減点対象ではなく、**エージェントの挙動傾向を把握する** ための分類。

| Form | 概要 |
| --- | --- |
| Form-1 | 既存ファイルへの変更を含む完全統合型 |
| Form-2 | 新規ファイルのみで完結する独立モック |
| Form-3 | React 非統合の HTML/CSS 静的プロトタイプ |
| Form-4 | コード生成なし（テキスト説明のみ）|

---

## 9. 想定成果物

- 実行結果生データ（各 run の diff / build ログ / judge 出力）
- 定量比較サマリ（エージェント別 / 入力別 / 仕様粒度別のスコア一覧）
- 定性分析（代表的な成功例・失敗例の抜粋）
- PoC 報告書（HTML 形式）:
  1. エグゼクティブサマリ
  2. 調査設計
  3. 主要結果（4 軸の比較）
  4. 定性分析
  5. 仕様駆動開発への示唆
  6. 制約・留意点
  7. 次フェーズ提案

---

## 10. 制約・留意点

- **エージェントの非決定性**: 各条件 1 run のため、確率的変動の完全排除は不可。代表的傾向把握に留まる。
- **Coding Agent ≠ IDE Chat**: GitHub Coding Agent 経由の結果は、IDE 内 Copilot Chat の体験と完全一致しない。結論適用時は前提明記が必要。
- **LLM-judge のバイアス**: D3・D4 は LLM による評価であり、judge モデルのバイアスを完全には排除できない。全 run で同一 judge を使い、一部は人手レビューで sanity check を行う。
- **コスト**: 48 run × 平均 5〜10 分で総計 5〜10 時間（並列実行可）、API 費用は数十ドル程度を想定。
