1エグゼクティブサマリ

結論 — 取り込む価値のある「LLM なし処理」

Understand-Anything は「AI がリポジトリを理解するツール」に見えますが、中身を精査すると、 調査の土台になる事実の収集はすべて AI を使わない機械的な処理(ルールベース)で行い、 AI は説明文の作成だけを担当しています。 この「機械でできることは機械でやる」部分こそが、今回のテスト支援エージェントに取り込むべき資産です。

取り込み候補は以下の 6 つです。上から順に「実装が軽く、テスト支援に直接効く」ものです。

#処理できることテスト支援への効果優先度
1リポジトリの棚卸し 全ファイルを一覧化し、言語・種類(コード/設定/ドキュメント等)・行数を機械的に判定 エージェントが調査を始める前に「正確な地図」を持てる。推測で探し回らなくなる 高
2テストと実装の対応付け ファイル名と配置のルール(例: foo.test.ts は foo.ts のテスト)から、どのテストがどの実装を検証しているかを特定 最も直接効く。「この変更に関係するテストはどれか」を AI の推測なしで答えられる 高
3フォルダ名からの役割推定 フォルダ名(controllers, models, tests など)から各ファイルの役割(API/画面/データ/テスト等)を推定 コードの「どのあたりを見ればよいか」の当たりを付けられる。ルール表を渡すだけでも効果あり 高
4ファイル間の依存関係の解決 「どのファイルがどのファイルを読み込んでいるか」を言語ごとのルールで正確に特定 変更点の周辺を正確に辿れる。対象言語を自社スタックに絞れば小さく作れる 中
5変更の影響範囲の算出 変更ファイルから依存関係を辿り、影響し得る箇所を提示 「どのテストを実行すべきか」の選定に直結 中
6再調査要否の自動判定 ファイルの構造変化だけを見て「調査結果を作り直す必要があるか」を判定 毎回ゼロから調査し直さなくなる。AI 利用料ゼロで判定できる 中

なぜ「LLM なし」の部分に価値があるのか

  • 正確AI にファイル一覧や依存関係を推測させると、間違えたり取りこぼしたりします(Understand-Anything の実測では、AI は依存関係の約 25% を取りこぼす)。機械的な処理なら取りこぼしゼロです。
  • 毎回同じ結果同じリポジトリに対して常に同じ結果が出ます(AI は実行のたびに答えが揺れる)。
  • コストゼロ・高速AI の API 利用料がかからず、数秒で終わります。
  • 検証可能ルールが明文化されているので、間違っていれば直せます。Understand-Anything はこの部分に 2,500 行超のテストを備えています。

取り込みの進め方(提案)

  1. まず優先度「高」の 3 つ(棚卸し・テスト対応付け・役割推定)を SKILL に組み込む。いずれも依存が薄く、単体で移植できる規模。
  2. 効果を確認後、優先度「中」(依存関係 → 影響範囲 → 再調査判定)を対象言語を絞って追加する。
  3. ライセンスは MIT のため、コードの流用・改変は可能。ただし丸ごと導入は実行環境の前提(Node 22 以上等)が重いため、必要な処理だけ移植する方が現実的。

2Understand-Anything では何が行われているのか

2.1 全体の仕組み

Understand-Anything は、リポジトリを解析して「知識グラフ」(ファイル・関数とそれらの関係を表した地図)を作るツールです。 処理は 7 段階のパイプラインで進みます。

  1. 準備 — 調査対象から外すファイル(ライブラリ置き場、ビルド成果物など)を決める機械
  2. 棚卸し — 全ファイルを列挙し、言語・種類・行数・依存関係を調べる機械
  3. 仕分け — 関係の深いファイル同士をグループにまとめ、AI が調査しやすい単位に分割する機械
  4. 調査 — グループごとに AI が「このファイルは何をしているか」の説明文を書くAI
  5. 検品 — AI の出力を機械的にチェックし、書式の揺れや間違いを自動修正する機械
  6. 整理 — アーキテクチャの層分けや「読む順番のガイド」を AI が作るAI
  7. 保存 — 結果を保存し、次回以降の差分更新に備えた「指紋」を取っておく機械

このうち AI が働くのは 4 と 6 だけです。それ以外はすべてルールベースの機械処理で、 さらに 4 と 6 の前後にも機械処理が挟まっています。

2.2 機械(ルールベース)が担っている 4 つの仕事

(1) 事実の収集 — ファイルの一覧、各ファイルの言語や種類、行数、「どのファイルがどのファイルを使っているか」、関数やクラスの一覧。 こうした「調べれば確実に分かる事実」は、すべて機械が収集します。 AI には「この事実を自分で調べ直すことを禁止する」という指示まで明文化されています。

(2) AI の下ごしらえ — 数百ファイルを AI に一度に渡すことはできないため、依存関係の近いファイル同士を数学的な手法(コミュニティ検出)でグループ化し、 「一緒に読むと理解しやすい単位」に分けてから AI に渡します。グループの大きさにも上限(35 ファイル)を設けて暴走を防いでいます。

(3) AI の出力の検品と修正 — AI の書いた結果には決まったパターンの間違いが混ざります。ID の書式が揺れる、「テスト → 実装」の関係の向きを逆に書く、 依存関係を約 25% 取りこぼす、などです。Understand-Anything はこれらを「よくある間違いのカタログ」として蓄積し、機械的に自動修正します。 AI を賢くしようとするのではなく、AI の出力を機械で直すという発想です。

(4) 再調査の要否判定 — コードが変更されたとき、各ファイルの「構造の指紋」(関数の名前や引数など、地図に影響する情報だけ)を取って前回と比較し、 「調査のやり直しが必要か / 一部だけ更新すればよいか / 何もしなくてよいか」を判定します。 この判定自体に AI は一切使わないため、変更のたびに AI 利用料が発生しません。

2.3 AI が担っている仕事

AI の担当は「人間向けの説明」に限定されています。各ファイルの要約文とタグ付け、アーキテクチャの層への命名、 初学者向けの「読む順番ガイド」の作成です。つまり 事実は機械、解釈と言語化は AI という分担です。

2.4 テスト支援エージェントへの示唆

エージェントの回答精度が低く見える原因の多くは、リポジトリの事実(何がどこにあり、何と繋がっているか)を 毎回 AI の推測で補っていることにあると考えられます。Understand-Anything の答えは明快で、 推測させる前に、正確な事実を機械が用意して渡すことです。 特に「テストと実装の対応付け」は、ファイル名のルールだけで機械的に導けるうえテスト支援の核心に直結するため、 最初に取り込む価値があります。

A.2 ignore 設定生成(generate-ignore.mjs / ignore-generator.ts / ignore-filter.ts)
  • .gitignore 互換の ignore npm ライブラリを使用。パターンは 3 層マージ(後勝ち、! 否定可): ①DEFAULT_IGNORE_PATTERNS(node_modules/ dist/ coverage/ 等のハードコード、ロックファイル、バイナリ拡張子、*.min.js *.generated.* 等)→ ②.understand-anything/.understandignore → ③ルートの .understandignore。
  • generator は .gitignore の取り込み提案、実在するテスト系ディレクトリ(EXACT_DIR_NAMES: __tests__ test tests fixtures testdata docs examples scripts migrations 等)、言語別テストパターン(JS/TS: *.test.*、Go: **/*_test.go、Java: **/src/test/**、C#: **/*Tests.cs 等)を全行コメントアウトで提案(自動有効化しない)。
A.3 プロジェクトスキャン(scan-project.mjs)
  1. 列挙: git ls-files -z -co --exclude-standard(NUL 区切りで非 ASCII パス対応)。git 不在時は再帰ウォーク(HARD_SKIP_DIRS = {node_modules, .git, .svn, .hg, __pycache__}、シンボリックリンク無視、localeCompare ソートで決定性確保)。
  2. 言語検出(detectLanguage): Dockerfile* 特別扱い → dotfile キー(.env.local→.env)→ 約 60 拡張子の LANGUAGE_BY_EXT → 未知拡張子はそのまま小文字で返す(null を返さない契約)→ ファイル名テーブル(Makefile 等)→ unknown。
  3. カテゴリ分類(detectCategory): LICENSE→code 例外 → infra ファイル名 → infra パス(.github/workflows/、/(^|\/)(k8s|kubernetes)\//)→ 拡張子テーブル(docs/config/infra/data/script/markup)→ code。
  4. 行数: バッファ内 0x0a バイト直接カウント。読取失敗は Warning + そのファイルのみドロップ。
  5. 複雑度: ファイル総数のみで ≤30: small / ≤150: moderate / ≤500: large / >500: very-large。

A.5 セマンティックバッチング(compute-batches.mjs)
  • importMap から graphology の無向グラフ(重みなし・code ファイルのみ)を構築し、graphology-communities-louvain でコミュニティ検出。失敗時はアルファベット順 12 ファイル/バッチに機械分割(algorithm: 'count-fallback')。
  • 定数: MAX_COMMUNITY_SIZE = 35(超過は均等分割)、MIN_BATCH_SIZE = 3 未満はプールして MAX_MERGE_TARGET = 25 ずつ misc バッチに再パック(87 個のサイズ 1 コミュニティ→87 回のサブエージェント起動になった実測が動機)。制御は件数のみで、トークン/行数見積もりは不使用。
  • 非 code ファイルは Groups A–E(Dockerfile+compose / GitHub Actions / GitLab+CircleCI / migrations SQL / 残りは親ディレクトリごと最大 20)。
  • neighborMap: 各ファイルの 1 ホップ近傍(import 先 + 被 import 元)のうち他バッチ分を export シンボル付きで最大 MAX_NEIGHBORS = 50 件記録(超過は次数降順)。
  • --changed-files モードは変更ファイルを含むバッチのみ出力、batchIndex は振り直さない(参照整合性)。
  • テストに「produces deterministic output across runs」ケースあり。
A.6 構造抽出(extract-structure.mjs + core extractor/parser 群)
  • 言語検出(language-registry.ts): ファイル名完全一致 → 拡張子一致の 2 段のみ。shebang・内容ベース検出は未実装(コメントに将来課題と明記)。builtin 約 40 言語。
  • フレームワーク検出(framework-registry.ts): マニフェスト内容の小文字化 + detectionKeywords の単純部分一致(includes())。10 フレームワーク。false positive の余地あり。layerHints/promptSnippetPath は定義のみで未配線。
  • コード言語(extractor 12 種): 正規表現ではなく tree-sitter AST 走査。WASM 文法ロード失敗言語は空結果でスキップし LLM に委譲。関数(名前・行範囲・パラメータ・戻り型。Python は self/cls 除外)、クラス(Go は struct/interface を写像し receiver からメソッド紐付け)、export 判定は言語慣習のコード化(Go: 先頭大文字、Python: トップレベル定義すべて)、call graph(関数スタック DFS)。
  • 非コードパーサ 12 種: ライブラリ+正規表現。YAML はパース失敗時に行正規表現 /^(\w[\w-]*)\s*:/ へフォールバック。Dockerfile は /^FROM\s+(\S+)(?:\s+[Aa][Ss]\s+(\S+))?/i 等。SQL は CREATE TABLE/VIEW/INDEX の正規表現 3 種。
A.7 マージ・修復(merge-batch-graphs.py, 約 1,165 行)

LLM 出力(batch-*.json)の系統的エラーを機械修復する「バリデータ/修復器」。

  1. ファイル名検証: batch-(\d+)(?:-part-(\d+))?\.json 非一致は除外して警告。マルチパート欠番も警告。
  2. ノード ID 正規化(4 段階): 二重プレフィックス除去 → プロジェクト名プレフィックス除去 → func:→function: 正準化 → 裸パスへ type からプレフィックス付与。
  3. complexity 正規化: 文字列エイリアス(low/easy→simple 等)+ 数値スケール(≤3→simple、≤6→moderate、7+→complex)。不明値は moderate + 要レビュー報告。
  4. エッジ参照書き換え → ノード重複排除(後勝ち)。
  5. tested_by リンカ(2 パス): LLM は方向を系統的に逆に書くため、エッジを捨てず方向だけ反転する。
    • テストファイル判定: JS/TS は stem が .test/.spec 終端、Go _test、Python test_*/*_test、Java *Test/*Tests/*IT、Kotlin/C# *Test/*Tests、C/C++ test_*/*_test。__tests__/helpers.ts のように命名が合わなければディレクトリ内でも非テスト扱い。
    • Pass 1: 端点分類し test→prod は反転+監査マーカー、test↔test/prod↔prod/orphan は drop。重複は weight 大が勝つ。
    • Pass 2: パス規約でペア補完(weight 0.5)。同ディレクトリ兄弟(foo.test.ts→foo.ts)→ __tests__ からの 1 段ウォークアウト → tests/ ルートから src/ app/ lib/ へのミラーツリー。Java は src/test/java→src/main/java、.NET は My.App.Tests→My.App 兄弟プロジェクトミラー。
    • カバレッジのある production ノードへ "tested" タグ。冪等性はテストで保証。
  6. エッジ重複排除: キー (source, target, type, direction)、weight 大勝ち。dangling エッジは drop してレポート。
  7. imports エッジ復元: scan-result.json の importMap を真実源に、LLM の取りこぼし(実測 ~25%)を weight 0.7・recoveredFromImportMap: true で再発行。

テスト: test_merge_batch_graphs.py(1,187 行・約 70 ケース)。

A.8 検証と多層修復(Phase 6 / schema.ts / normalize-graph.ts)
  • Phase 4/5 の LLM 出力には決定的正規化手順を規定(エンベロープ剥がし、フィールド名変更、file: プレフィックス付与、dangling 除去、order ソート)。
  • Phase 6 デフォルト検証はインライン Node スクリプト(非LLM): 必須フィールド、重複 ID、エッジ参照整合性、レイヤ網羅性、孤児検出、統計。--review 時のみ LLM レビュアー。
  • schema.ts(zod)の多層修復: null 除去・小文字化 → エイリアス正規化(ノード型 約 50 種、エッジ型 約 40 種)→ auto-fix(欠損 type→"file"、weight 欠損→0.5、[0,1] clamp)→ 個別検証で不正要素のみ drop → 有効ノード 0 件のみ fatal。
A.9 増分更新(fingerprint.ts / change-classifier.ts / hooks)
  • fingerprint: SHA-256 contentHash + tree-sitter 由来のシグネチャ(関数名・パラメータ・戻り型・export・行数、クラス構成、import/export 一覧)。関数本体はハッシュしない。
  • 変更分類: NONE(ハッシュ一致)/ COSMETIC(内容変化だが構造一致)/ STRUCTURAL(構造変化、または行数比 1.5 倍超 or 0.5 倍未満)。tree-sitter 非対応言語は常に STRUCTURAL(保守的)。
条件アクション
構造変更 0 件SKIP(LLM トークン消費ゼロ)
構造変更 >30 件 または グラフ内ファイルの >50%FULL_UPDATE
構造変更 >10 件 または 新トップレベルディレクトリ出現ARCHITECTURE_UPDATE
それ以外PARTIAL_UPDATE(変更ファイルのみ再解析)
  • hooks: PostToolUse で grep -qE 'git\s+(commit|merge|cherry-pick|rebase)' 判定、SessionStart でコミットハッシュ比較。auto-update の Phase 0–1 はゼロトークンで、SKIP/FULL 判定ならそこで停止。
  • 反映(staleness.ts mergeGraphUpdate): 旧ノード削除 → dangling エッジ削除 → 追加 → メタ更新。
A.10 その他のルールベース機能
  • レイヤ検出(layer-detector.ts): パスセグメントを LAYER_PATTERNS(先勝ち・複数形許容)と照合 — API(routes controller handler endpoint api)、Service(service usecase business)、Data(model entity schema repository migration)、UI(component view page screen)、Middleware、External、Background(worker job queue cron)、Utility(util helper lib common shared)、Test、Config。不一致は "Core"。
  • diff 影響分析(src/diff-analyzer.ts): 変更ファイル → ノードマップ → contains で関数/クラス展開 → 1 ホップ近傍を blast radius として収集。リスク判定はハードコード規則(complex ノードあり / 影響レイヤ >1 / 影響ノード >5 / 未マップあり / それ以外 Low risk)。
  • 検索(search.ts): fuse.js ファジー検索(重み name 0.4 / tags 0.3 / summary 0.2 / languageNotes 0.1、threshold 0.4)。semantic 検索(embedding-search.ts)は受け皿のみで embedding 生成コードは不在。ダッシュボードは常に fuzzy にフォールバック。
  • サブドメイングラフ統合(merge-subdomain-graphs.py): ID 後勝ち、エッジ (source, target, type) キー + weight 大勝ち。layers は和集合、tour はタイトルキー結合。
  • ドメイン文脈抽出(extract-domain-context.py): エントリポイント検出の正規表現 11 種(Express/Flask/FastAPI/Spring ルート、Next.js HTTP export、CLI、イベント、cron、GraphQL、gRPC)、パスキーワード 25 語のスコアリングで上位 40 ファイルのシグネチャ抽出、出力 512KB 上限。
  • ナレッジベース解析(parse-knowledge-base.py): wikilink 正規表現 \[\[([^\]|]+)(?:\|([^\]]+))?\]\]、wikilink 密度で complexity 算出(>15=complex、>5=moderate)。
A.11 横断的な設計思想
  1. 事実と解釈の分離: 機械的に取れる事実は 100% 決定的スクリプトが生成し、LLM には再導出を禁止。LLM は要約・タグ・命名・低確度の意味的エッジのみ。
  2. LLM 出力の決定的修復層: 系統的エラーをカタログ化して機械修復。「LLM を賢くする」のではなく「出力を機械で直す」。
  3. ゼロトークンのゲート: 変更検知と更新分類を完全ルールベースにし、LLM を呼ぶかどうか自体を決定的に判断。
  4. 劣化継続: throw せず Warning + 空値で継続(per-file resilience)。フォールバックも決定的。
  5. 決定性の徹底: 全出力ソート済み・再実行時バイト一致。テスト 2,500 行超が閾値を固定化。
  6. ディスパッチ単位の設計: LLM 呼び出し粒度を import グラフのコミュニティ構造で決め、件数定数(35/25/20/3)で暴走を防止。
