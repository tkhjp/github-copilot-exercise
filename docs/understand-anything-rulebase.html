1. Tech leader 向け要点

Understand-Anything は「LLM が全部読む」ツールではありません。 実態は、機械処理で repo の事実を作り、LLM が説明や構造化を補い、最後に機械処理で検証・修復する pipeline です。

INPUT 入力は repository path

/understand [path] が起点。flags により full rebuild、auto-update、language などを制御する。

PROCESS 事実、解釈、修復を分ける

file list/import/structure は rule-based。summary/layer/tour は LLM。ID/edge/validation は再び rule-based。

OUTPUT 出力は graph と補助ファイル

knowledge-graph.json、fingerprints、meta、必要に応じて domain-graph.json を作る。

組内に説明するなら:

Understand-Anything は、repo の「事実」を決定的 script で集め、その事実を LLM に渡して説明を作らせ、 LLM の出力を再び決定的 script で修復・検証して graph にする仕組みです。

したがって見るべきポイントは UI ではなく、どの情報が rule-based の事実で、どの情報が LLM の解釈なのか、 そして source code や生成物をどう安全に扱うかです。

2. LLM なしでできる範囲

LLM を完全に外しても、Understand-Anything のかなり大きな部分は動かせます。 ただし、その出力は「人間向けの理解資料」ではなく、決定的に作れる repository fact map になります。

結論

LLM なしで到達できるのは、構造・依存・変更・候補関係の machine-readable graph までです。 「このファイルはなぜ重要か」「この product の business flow は何か」「初心者はどこから読むべきか」といった説明・解釈は弱くなります。

LLM なし版の成果物イメージ

LLM なしで作るなら、最終成果物は knowledge-graph.json というより、 repository-facts.json のようなものになります。中身は file、function、class、import、tested_by 候補、 entry point 候補、fingerprint、変更分類です。人が読む説明資料ではなく、他の LLM/test framework に渡すための 事実ベースの context と考えるのが自然です。

3. 安全・制約

「コードを読むだけ」でも、source path、関数名、domain 用 snippet などが生成物や LLM context に入ります。 安全性は full pipeline と rule-based 部分で分けて見る必要があります。

観点現状説明すべき責任/制約
読み取り範囲 projectRoot 配下を読む。.understandignore と既定除外を適用。 root を許可 workspace に閉じる。任意絶対パスや .. を処理 API で通さない。
コード実行 解析処理は source を読むだけ。test/build/package script は実行しない。 解析処理と実行処理を分ける。解析側で project scripts を起動しない。
外部コマンド git ls-files を固定 argv で実行。失敗時は walk fallback。 shell string に user input を連結しない。任意 command 実行にしない。
LLM 送信 full /understand では README、manifest、file structure、batch context、必要に応じて snippets が LLM に渡る。 秘密情報を含む repo では実行環境/送信先の確認が必要。rule-based only ならこの送信は避けられる。
生成物の公開 graph には path、関数名、クラス名、summary、domain 語彙が入る。 公開 demo では raw source preview、intermediate、domain-context を出さない。必要なら redaction。
License Understand-Anything は MIT License。 切り出し/改変は可能。著作権表示と MIT 許諾文を同梱。依存 license は別途棚卸し。

重要: rule-based script だけを使う場合と、full pipeline で LLM agents を使う場合では、情報の外部送信リスクが違います。この差を説明資料内で明確に分けます。

summary、tags、architecture/tour を含む graph を保存する。便利だが、公開 demo では情報量が増える点に注意。

構造情報と fingerprint は保存できる。外部送信を避けやすいが、UI 上の説明文や business 解釈は薄くなる。

処理コンポーネントごとの機能と実装方法

Understand-Anything には、MCP tool や function calling のような一等公民としての Tool 定義はありません。 ここでは説明上の処理単位として、Skill / Agent / Script / Core API / Dashboard をまとめて「処理コンポーネント」と呼びます。 Rule-based は機械的に source/metadata を読む処理、LLM は意味づけや説明生成、Mixed は script で事実を作って LLM が解釈する処理です。

domain-analyzer LLM business domain、flow、step を graph 化する。 既存 knowledge-graph.json があれば graph から推論。なければ domain-context.json を LLM に渡して分析する。 knowledge graph または domain context → domain-graph.json

5. 技術スタック

        

          処理は UI と解析 pipeline に分かれます。LLM なしで使える rule-based 部分は主に Node.js/TypeScript、tree-sitter、Python script で構成されています。
        

        

          

            
領域技術使われ方LLMなし利用

            
              

                実行基盤
                Node.js 22 以上、pnpm 10 系、TypeScript
                scan、import 解決、構造抽出、fingerprint、更新分類。
                必要
              

              

                構文解析
                web-tree-sitter + tree-sitter grammar 群
                functions/classes/imports/exports/callGraph を AST から抽出。
                必要

                YAML parser、正規表現 parser、zod schema

                必要

                import graph から community を検出し、解析 batch を作る。
                必要

                必要

6. 責任境界

        

          この仕組みを説明するときは、「誰が何を保証しているか」を分けて話す必要があります。
        

        

          

            
Orchestrator / SKILL.md

            
処理順、入力/出力パス、subagent dispatch、review/cleanup の手順を管理する。root と実行モードの責任を持つ。

          

          

            
Deterministic scripts

            
file list、importMap、structure、batch、merge、validation、fingerprint を作る。同じ入力に対し同じ結果を返す責任を持つ。

          

          

            
LLM agents

            
summary、tags、architecture layers、tour、domain flow など、人間向けの説明と推論を担当する。解釈なので検証対象。

          

          

            
Operator / user

            
対象 repo、除外ルール、公開範囲、秘密情報の扱い、license notice の確認に責任を持つ。

          

        


        
どの出力をどう扱うか

        

          

            
出力性質扱い方

            
              

                files[]、importMap、fingerprints
                機械的事実
                比較的信用できる。parser/resolver の対応範囲外だけ注意。
              

              

                summary、tags、layers、tour
                LLM 解釈
                説明材料として使う。重要判断には fact edge と source を併用する。
              

              

                tested_by、import edges after merge
                LLM + rule repair
                direction や missing import は補正される。semantic な「本当に検証しているか」は別途確認。
              

              

                domain-graph.json
                domain 推論
                business 説明には便利。ただし rule-based だけで生成されたものではない。

8. 技術根拠

        

          実装担当が詳細を確認するための折りたたみメモです。
        

        

          
          
        


        

          scan-project.mjsfile enumeration と分類
          

            
    
              
  • git ls-files -z -co --exclude-standard を優先。git 不在時は recursive walk。
    • 
              
  • hard skip: node_modules、.git、__pycache__ 等。symlink は追わない。
    • 
              
  • detectCategory(): LICENSE 例外、infra filename/path、拡張子 table、fallback code。
    • 
            

          

        


        

          extract-import-map.mjs言語別 import resolver
          

            
    
              
  • tree-sitter で raw import を抽出し、project 内部 file set に解決できたものだけ採用。
    • 
              
  • TS/JS、Python、Go、Java/Kotlin/C#、Ruby、PHP、Rust、C/C++、Swift に言語別 rule。
    • 
              
  • tree-sitter 初期化失敗時は importMap を空にして継続。
    • 
            

          

        


        

          merge-batch-graphs.pyLLM output repair
          

            
    
              
  • node ID 正規化、complexity 正規化、dangling edge drop、重複 edge dedupe。
    • 
              
  • tested_by は LLM が逆向きに出した test → production を production → test に反転。
    • 
              
  • importMap を真実源として、LLM の import edge 取りこぼしを再発行。
    • 
            

          

        


        

          fingerprint.ts / change-classifier.ts増分更新判断
          

            
    
              
  • fingerprint は contentHash と構造 signature。関数名、引数、戻り型、class、import/export を比較。
    • 
              
  • COSMETIC は内容変化だが構造同一。tree-sitter 非対応は保守的に structural。
    • 
              
  • 構造変更 0 件は SKIP、大量変更は FULL_UPDATE。
    • 
            

          

        


        

          extract-domain-context.pyBusiness/domain 前処理
          

            
    
              
  • HTTP route、CLI、event、cron、GraphQL、gRPC、exported handler を正規表現で検出。
    • 
              
  • 候補 file signature は business logic らしい path keyword で優先付け。
    • 
              
  • 最終 domain graph は LLM domain-analyzer が生成する。
