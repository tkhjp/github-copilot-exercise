### 1. 全コンポーネント一覧

このシステムは、Webアプリケーションのフロントエンドからデータベースまでを構成する典型的な3層アーキテクチャを採用しています。

**【外部/エッジ層】**
* **Route 53 DNS:** ドメイン名解決サービス。トラフィックを適切なエントリーポイントに誘導します。
* **CloudFront CDN:** コンテンツデリバリーネットワーク。静的コンテンツの配信速度向上と負荷分散を行います。
* **S3 Static Assets:** 静的ファイル（画像、CSSなど）のストレージ。

**【ロードバランシング/インフラ層】**
* **ALB (Application Load Balancer):** アプリケーション層へのトラフィックを振り分けるL7ロードバランサー。各AZに配置されています。
* **NAT Gateway:** プライベートサブネット内のリソースがインターネット上の外部サービス（例：パッチ適用、外部APIコール）へアウトバウンド通信を行うためのゲートウェイ。

**【アプリケーション層 (Compute)】**
* **ECS Fargate App Service:** コンテナ化されたWebアプリケーションを実行するコンピューティングサービス。各AZに配置されています。
* **ElastiCache Redis:** アプリケーションのキャッシュ層として機能します。各AZに配置されています。

**【データ層 (Persistence)】**
* **RDS PostgreSQL Primary:** メインのデータベースインスタンス（プライマリ）。AZ-aに配置されています。
* **RDS PostgreSQL Read Replica:** 読み取り専用レプリカ。AZ-bに配置されており、リード負荷を分散します。

**【監視・管理】**
* **IAM Roles:** AWSリソースへのアクセス権限を定義するセキュリティ機構。
* **CloudWatch Logs/Metrics:** システムのログ収集とパフォーマンスメトリクスの監視に使用されます。

### 2. VPC / AZ / サブネットの階層構造

このアーキテクチャは、単一のVPC内に複数のアベイラビリティゾーン（AZ）を跨いでリソースを配置することで高可用性を実現しています。

**【VPC】**
* **CIDR:** `10.0.0.0/16` (図示されている範囲)
* **目的:** すべてのAWSリソースを論理的に隔離するネットワーク境界です。

**【AZとサブネットの階層（冗長化構造）】**

この設計は、2つのアベイラビリティゾーン（AZ-a と AZ-b）にわたって、パブリック層とプライベート層を分離しています。

| ゾーン | サブネットタイプ | 配置されている主要コンポーネント | 役割 |
| :--- | :--- | :--- | :--- |
| **AZ-a** | **Public Subnet (App)** | ALB, NAT GW (AZ-a) | インターネットからの直接アクセスを受け付ける層。NAT経由でプライベートリソースへ接続する出口を提供。 |
| **AZ-a** | **Private Subnet (App)** | ECS Fargate (AZ-a), ElastiCache Redis (AZ-a), RDS Primary | アプリケーションロジックとメインDBが稼働する、インターネットから直接到達できない安全な層。 |
| **AZ-b** | **Public Subnet (App)** | ALB, NAT GW (AZ-b) | AZ-aと同様の役割を担い、トラフィックを受け入れます。 |
| **AZ-b** | **Private Subnet (App)** | ECS Fargate (AZ-b), ElastiCache Redis (AZ-b), RDS Read Replica | アプリケーションロジックとリードレプリカDBが稼働する安全な層。 |

### 4. Multi-AZ による冗長化の構造

このアーキテクチャは、単一障害点 (Single Point of Failure: SPOF) を排除し、高い可用性（High Availability）を確保するように設計されています。

**【冗長化のポイント】**

1. **ロードバランシングによる耐障害性:**
    * **ALB** がAZ-aとAZ-bの両方に配置されているため、いずれか一つのAZでALBがダウンしても、もう一方のAZのALBがトラフィックを受け取り続けることができます。
2. **アプリケーション層の冗長化 (Compute):**
    * **ECS Fargate App Service** がAZ-aとAZ-bの両方にデプロイされています。これにより、特定のAZでインスタンス全体が停止しても、もう一方のAZのコンテナ群がリクエストを処理し続けます。
3. **データ層の冗長化 (Persistence):**
    * **RDS PostgreSQL** は、プライマリ（AZ-a）とリードレプリカ（AZ-b）という形で配置されています。
        * **読み取り負荷分散:** アプリケーションは読み取りクエリをレプリカに振り分けることで、プライマリDBの負荷を軽減します。
        * **フェイルオーバー耐性:** プライマリインスタンスが障害を起こした場合、RDSの自動フェイルオーバー機能により、リードレプリカ（または別のスタンバイインスタンス）が新しいプライマリとして昇格し、サービスの中断時間を最小限に抑えます。
4. **ネットワーク層の冗長化:**
    * **NAT Gateway** がAZ-aとAZ-bの両方に存在するため、片方のAZでNAT GWが利用できなくなっても、もう一方のAZ経由でアウトバウンド通信を継続できます。

**【まとめ】**
この設計は、「**複数のアベイラビリティゾーンにわたって、ロードバランサ、アプリケーションコンテナ、およびデータベースインスタンス（プライマリとレプリカ）を分散配置する**」という標準的かつ堅牢なマルチAZ戦略を採用しています。
