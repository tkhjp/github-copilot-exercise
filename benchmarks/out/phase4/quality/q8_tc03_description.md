## 1. 全コンポーネント一覧

このアーキテクチャに含まれる主要なコンポーネントは以下の通りです。

**インフラストラクチャ/ネットワーク層:**
* **VPC (Virtual Private Cloud):** すべてのリソースをホストする論理的なプライベートネットワーク。
* **AZ (Availability Zone):** 複数の物理的に分離されたデータセンター（この図では `ap-northeast-1a` と `ap-northeast-1c` の2つのAZが使用されています）。
* **サブネット:** VPCをさらに分割した論理的なセグメント。
    * **Public Subnet (パブリックサブネット):** インターネットゲートウェイ経由で外部と通信できるリソース（ALB, NAT Gatewayなど）を配置します。
    * **Private Subnet (プライベートサブネット):** インターネットから直接アクセスできない、アプリケーションやデータベースなどの機密性の高いリソースを配置します。
    * **DB Subnet:** データベースインスタンス専用のサブネット。

**サービス層（AWSマネージドサービス）:**
* **Route 53 DNS:** ドメイン名の名前解決を行うDNSサービス。
* **CloudFront CDN:** コンテンツ配信ネットワーク。静的コンテンツをエッジロケーションから高速に配信します。
* **S3 Static Assets:** 静的なウェブアセット（画像、CSSなど）をホストするストレージサービス。
* **ALB (Application Load Balancer):** アプリケーション層のトラフィックを複数のターゲット（EC2/ECSタスク）に分散させるロードバランサー。パブリックサブネットに配置されます。
* **NAT Gateway:** プライベートサブネット内のリソースがインターネット上の外部サービスへアウトバウンド通信を行うためのゲートウェイ。パブリックサブネットに配置されます。
* **RDS PostgreSQL (Primary/Read Replica):** リレーショナルデータベースサービス。プライマリとリードレプリカで構成され、高可用性を実現しています。

**アプリケーション層（コンテナ化）:**
* **ECS Fargate App Service:** コンテナ化されたアプリケーションを実行するサーバーレスコンピューティング環境。プライベートサブネットに配置されます。
* **ElastiCache Redis:** キャッシュサービスとして利用されます。プライベートサブネットに配置されます。

**管理・監視層:**
* **IAM Roles:** AWSリソースへのアクセス権限を定義します。
* **CloudWatch Logs/Metrics:** アプリケーションやインフラストラクチャのログ収集とメトリクス監視を行います。

## 2. VPC/AZ/サブネットの階層構造

このアーキテクチャは、**マルチAZ構成を採用したVPC内での論理的な分離**が特徴です。

### 階層構造
1. **VPC (Virtual Private Cloud):** 全体の境界線（`vpc: 10.0.0.0/16`）。
2. **Availability Zones (AZs):** 物理的に独立したデータセンターのグループ分け。
    * AZ-a (`ap-northeast-1a`)
    * AZ-c (`ap-northeast-1c`)
3. **サブネット:** VPCをAZごとに論理的に分割し、役割（パブリック/プライベート）を定義します。

### サブネットの配置と役割分担
| 役割 | 配置されるコンポーネント例 | AZごとの配置 | 特徴 |
| :--- | :--- | :--- | :--- |
| **Public Subnet** | ALB, NAT Gateway | AZ-a および AZ-c にそれぞれ存在 | インターネットからの直接アクセスを受け付ける層。NAT GWはプライベートリソースの外部通信を仲介する。 |
| **Private Subnet (App)** | ECS Fargate App Service, ElastiCache Redis | AZ-a および AZ-c にそれぞれ存在 | アプリケーションロジックが動作する層。インターネットからの直接アクセスは遮断される。 |
| **DB Subnet** | RDS PostgreSQL (Primary/Replica) | AZ-a および AZ-c にそれぞれ存在（※図の表現による） | データベース専用のセグメント。通常、プライベートサブネット内に配置され、セキュリティを確保する。 |

## 3. 主要な接続とプロトコル

主要なデータフローと使用される通信プロトコルは以下の通りです。

| 送信元 $\rightarrow$ 受信先 | 経路/コンポーネント | プロトコル (推測) | 説明 |
| :--- | :--- | :--- | :--- |

| **ALB $\rightarrow$ ECS Fargate** | ALB $\rightarrow$ Private Subnet | HTTP/HTTPS (TCP) | ロードバランサーがアプリケーションコンテナ群へトラフィックを振り分けます。 |

## 4. Multi-AZ による冗長化の構造

このアーキテクチャは、単一障害点（SPOF）を排除し、高い可用性（High Availability: HA）を実現するために、**複数のアベイラビリティゾーン (AZ) にわたってコンポーネントを分散配置する**ことで冗長性を確保しています。

### 冗長化の具体的な構造
1. **ロードバランシングによるトラフィック分散:**
    * **ALB:** AZ-a と AZ-c の両方のパブリックサブネットに存在し、リクエストを複数のAZにまたがって受け付けます。これにより、特定のAZ全体がダウンしてもサービスは継続します。

2. **アプリケーション層の冗長化 (ECS Fargate):**
    * ECSタスク（Fargate）は、AZ-a と AZ-c の両方のプライベートサブネット内にデプロイされています。もしAZ-aで障害が発生した場合、ALBは自動的に健全なAZ-cのインスタンスへトラフィックをルーティングします。

3. **データベース層の冗長化 (RDS):**
    * **Primary/Read Replica構成:** RDS PostgreSQLが使用されており、プライマリインスタンスとリードレプリカが異なるAZに配置されている可能性が高いです（図では両方のAZでDBサブネットが存在するため）。これにより、プライマリインスタンスが利用不能になった場合、自動フェイルオーバーによりサービスを継続できます。

4. **ネットワーク層の冗長化:**
    * **NAT Gateway:** 各AZにNAT Gatewayを配置することで、片方のAZのゲートウェイに障害が発生しても、もう一方のAZ経由でアウトバウンド通信が維持されます。

**【まとめ】**
この設計は、「**各レイヤー（ロードバランサー $\rightarrow$ アプリケーション $\rightarrow$ データベース）において、最低2つの異なる物理的な場所（AZ）にコンポーネントを複製し、自動フェイルオーバー機構を利用する**」という、クラウドにおける標準的かつ堅牢な高可用性設計パターンを採用しています。
