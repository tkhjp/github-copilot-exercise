# 主要な業務インサイト

text_vs_image 自動評価テスト (`gemini-3.1-pro-preview`, 2026-04-09) の結果から得られた、業務で使う際の主要な示唆。

## 1. pptx 丸ごとを image として渡すユースケースでは image 直接が +33pp 圧勝
text 化では 1 枚の密度を吸収しきれない。
- 参考: **tc01 混合スライド** (extraction) で Case 1 = 52% / Case 2 = 85% / 差 **+33pp**
- 1 枚に flow + chart + table + code が同居するような密な画像は、specialized 記述を経由すると情報が圧縮されて細部が落ちる
- **結論**: pptx スライド丸ごとの分析業務では、画像アップロードが使える環境の方が本質的に有利

## 2. RFP のような「要素→指示」型の構造化タスクでは適切な specialized プロンプトが image を超える
- 参考: **tc02 UI 変更要求** (extraction) で Case 1 = 68% / Case 2 = 48% / 差 **−20pp** (text が逆転勝ち)
- `ui_change.md` プロンプトが「対象 UI 要素 → 変更内容」表形式を強制した結果、構造的に漏れの少ない記述が生成された
- 画像直接の Case 2 は自由形式回答なので、吹き出しと対象要素の対応づけを一部取りこぼした
- **結論**: 構造化された出力が要求される業務（RFP 読解、UI 仕様書レビュー等）では、specialized プロンプト + text 経路が安定している場合がある

## 3. judgment タスクでも image/text の差は中程度
画像直接が有利な傾向はあるが決定的ではない。
- 参考:
  - **tc02_judge** (UI 工数見積) — Case 1 = 75% / Case 2 = 88% / 差 **+13pp**
  - **tc03_judge** (AWS 視覚特徴読解) — Case 1 = 95% / Case 2 = 95% / 差 **0pp**
- 「画像を見て判断する」系のタスクでも、specialized 記述が十分に構造化されていれば text 経由でもほぼ同等の判断が可能
- **結論**: extraction で情報が text に十分に残せているなら、その後の判断品質はあまり劣化しない

## 4. Case 1 プロンプトの文言がスコアを支配
「推測禁止」が強すぎると LLM は「判断不可能」に逃げる。
- 事例: tc02_judge の Case 1 初回実行で全 16 facts が `missing` (**0%**)
  - 原因となった文言: 「記述に書かれていない情報は推測しないでください」
  - 結果: LLM は「工数情報は記述に無い → 回答不可」と答えを拒否
- 対応: 文言を「画像の事実は記述を根拠にし、判断には一般知識を使って OK」に緩和
  - スコアが **0% → 75%** に急回復
- **結論**: Copilot Chat のような業務ツールで text 経由の判断を求める場合、プロンプト側で「ドメイン知識の使用を明示的に許可」する必要がある。過度の制約は LLM を沈黙させる

## 5. describe 列は judgment ケースの採点対象にならない
合計値は参考程度に留めるべき。
- 事例:
  - tc02_judge: Generic 記述 = 13% / Specialized 記述 = **0%**
  - 理由: specialized 記述は UI 要素を列挙するだけで、「工数見積もり」のような judgment 答えは含まない
- レポートの集計サマリー (全列平均) は extraction と judgment を混在させているので、**列間の直接比較には向かない**
- **結論**: 評価軸を設計する際は「記述列」と「回答列」を別の ground truth で評価すべき。合計値ではなく、ケース単位・列単位の差分を見るのが正しい

## 既知の限界

- 全列を同じ Gemini モデルで生成しているため、**自己参照バイアス** が入り得る
- **LLM-as-judge の非決定性** — 再実行で数 pp スコアが揺れる
- 5 ケースは **探索的テスト** であり統計的有意性はない
- extraction と judgment は ground truth の性質が異なるので、**合計列の単純比較は誤解を招く**
