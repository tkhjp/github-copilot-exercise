# 主要な業務インサイト

text_vs_image 自動評価テスト (`gemini-3.1-pro-preview`、2026-04-09 実施) の結果から得られた、業務利用における主要な示唆をまとめる。

## 1. スライドを丸ごと画像として渡すユースケースでは、画像を直接渡す方が +33pp で大きく上回った

テキスト記述を経由する方式では、1 枚のスライドに詰め込まれた情報量を十分にカバーしきれない。

- 参考: **tc01 混合スライド** (extraction) で Case 1 = 52% / Case 2 = 85% / 差 **+33pp**
- 1 枚の中に flow + chart + table + code が同居するような高密度な画像では、specialized 記述を経由する過程で情報が圧縮され、細部が落ちやすい。
- **結論**: pptx スライドを丸ごと扱う分析業務では、画像アップロードが使える環境の方が明らかに有利。

## 2. RFP のような「要素 → 指示」型の構造化タスクでは、適切な specialized プロンプトが画像直接を上回ることがある

- 参考: **tc02 UI 変更要求** (extraction) で Case 1 = 68% / Case 2 = 48% / 差 **−20pp** (テキスト経由が逆転)
- `ui_change.md` プロンプトが「対象 UI 要素 → 変更内容」という表形式で出力するように指示しているため、構造的に漏れの少ない記述が生成された。
- 一方で画像を直接渡した Case 2 は自由記述の形で回答するため、吹き出しと対象要素の対応づけを一部取りこぼしている。
- **結論**: 構造化された出力が求められる業務（RFP 読解、UI 仕様書レビュー等）では、specialized プロンプト + テキスト経路の方がかえって安定する場合がある。

## 3. judgment タスクでも image / text の差は限定的

画像直接の方が有利な傾向はあるものの、決定的な差ではない。

- 参考:
  - **tc02_judge** (UI 工数見積) — Case 1 = 75% / Case 2 = 88% / 差 **+13pp**
  - **tc03_judge** (AWS 視覚特徴読解) — Case 1 = 95% / Case 2 = 95% / 差 **0pp**
- 「画像を見て判断する」系のタスクであっても、specialized 記述が十分に構造化されていれば、テキスト経由でもほぼ同等の判断品質が得られる。
- **結論**: extraction 段階でテキストに十分な情報が残せているなら、その後の判断品質はほとんど劣化しない。

## 4. Case 1 のプロンプト文言がスコアを大きく左右する

「推測禁止」の制約を強くし過ぎると、LLM は「判断できない」として回答を拒否してしまう。

- 事例: tc02_judge の Case 1 初回実行で、全 16 件のファクトがすべて `missing` と判定され、**スコアが 0%** になった。
  - 原因となった文言: 「記述に書かれていない情報は推測しないでください」
  - 結果: LLM は「工数情報は記述に含まれていないため回答できない」として、すべての設問に対して判断を放棄した。
- 対応: 文言を「画像に関する事実は記述を根拠にし、判断や推論には一般的なドメイン知識を自由に使ってよい」に緩和した。
  - スコアが **0% → 75%** まで急回復した。
- **結論**: Copilot Chat のような業務ツールでテキスト経由の判断を求める際は、プロンプト側で「ドメイン知識の使用を明示的に許可する」ことが不可欠。制約が強過ぎると、LLM は回答自体を避けてしまう。

## 5. describe 列は judgment ケースの採点対象として成立しない

合計スコアは参考程度に留めて読むべき。

- 事例:
  - tc02_judge: Generic 記述 = 13% / Specialized 記述 = **0%**
  - 理由: specialized 記述は UI 要素を列挙するだけであり、「工数見積もり」のような judgment 系の答えはそもそも含まれない。
- レポートの集計サマリー (全列平均) では extraction と judgment が混在しているため、**列間の単純な比較には向かない**。
- **結論**: 評価軸を設計する際は「記述列」と「回答列」で別々の ground truth を用意するのが望ましい。分析する際は合計値ではなく、ケース単位・列単位の差分に注目するべき。

## 既知の限界

- 全列を同じ Gemini モデルで生成しているため、**自己参照バイアス** が入り込む可能性がある。
- **LLM-as-judge の非決定性** により、再実行するたびに数 pp 程度スコアが揺れる。
- 5 ケースという規模は **探索的な検証** に留まり、統計的な有意性を主張できる水準ではない。
- extraction と judgment では ground truth の性質が異なるため、**合計列での単純比較は誤解を招きやすい**。
