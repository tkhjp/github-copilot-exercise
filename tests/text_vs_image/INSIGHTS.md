# 主要な業務インサイト (v2)

text_vs_image 自動評価テスト v2 の結果から得られた、業務利用における主要な示唆をまとめる。

v2 では describe (画像→テキスト記述) に Gemini Flash 系、answer (質問への回答) と judge (採点) に GPT-5.4 を使い、モデルを分離することで自己参照バイアスを低減している。

## 全体スコア

### Phase 1: Describe モデル比較 (Gemini, extraction ケースのみ)

```
                    generic     specialized
gemini-3-flash       95.5%       94.6%
gemini-3.1-flash-lite 93.0%      93.2%
```

→ **gemini-3-flash を採用** (combined avg 95.1% vs 93.1%)

### Phase 2: Answer 比較 (GPT-5.4, extraction + judgment)

| ケース | タイプ | A1 text→GPT | A2 img→GPT | A2−A1 |
|---|---|---|---|---|
| tc01 混合スライド | extraction | **95.8%** | 91.7% | −4.2pp |
| tc02 UI 変更要求 | extraction | 77.5% | **87.5%** | +10.0pp |
| tc03 AWS アーキ | extraction | 90.4% | 90.4% | 0pp |
| tc04 テキストのみ | extraction | **100%** | **100%** | 0pp |
| tc02_judge UI 工数見積 | judgment | 85.7% | 85.7% | 0pp |
| tc03_judge 視覚特徴 | judgment | **88.9%** | 77.8% | −11.1pp |

---

## 1. テキストのみの画像は、テキスト経由でも情報損失ゼロ

tc04 (仕様書ページのスクリーンショット、図表を含まない) で A1 = A2 = 100% を確認した。

- Gemini の describe がテキストを正確に転写し、GPT-5.4 がそれを読んで全 22 facts を正しく回答。
- 画像を直接渡した場合と**完全に同じスコア**。
- **結論**: 画像に図表やチャートが含まれず、テキストのみの場合は、テキスト記述経由で十分に代替できる。Copilot Chat の画像機能が使えない環境でも、テキスト中心の文書は describe スクリプトで問題なく対応可能。

## 2. 密度の高い混合スライドでも、質の良い describe があれば text 経由が勝つことがある

tc01 (1 枚にフロー図+棒グラフ+表+コードを同居) で A1 = 95.8%、A2 = 91.7%。

- v1 (全列 Gemini) では image 直接が +33pp で圧勝していたが、v2 では text 経由が −4.2pp で逆転。
- 理由: Gemini Flash の describe 品質 (D1=97.9%, D2=97.9%) が十分に高く、GPT-5.4 がその記述から高精度に回答できたため。
- v1 との差は**回答モデルの違い** (Gemini 3.1 Pro vs GPT-5.4) によるものが大きい。
- **結論**: describe の品質が高ければ、text 経路でも image 直接と同等以上の結果が得られる。**describe プロンプトの品質が鍵**。

## 3. 空間的な対応づけが必要なタスクでは、依然として image 直接が有利

tc02 (UI モックアップ + 赤い吹き出しの変更要求) で A1 = 77.5%、A2 = 87.5% (+10pp)。

- 「吹き出しがどの UI 要素を指しているか」の空間的対応づけは、テキスト記述だけでは伝えにくい。
- specialized プロンプト (`ui_change.md`) が `{対象要素 → 変更内容}` 表を生成しているが、それでも GPT が text から spatial relationship を再構成するのには限界がある。
- **結論**: RFP やアノテーション付きスクリーンショットのような「空間的な指示が重要」なタスクでは、画像を直接渡せる環境の方が安全。

## 4. 構造的なアーキテクチャ図は、Mermaid 化すれば text 経由で十分

tc03 (AWS Multi-AZ、12+ コンポーネント) で A1 = A2 = 90.4%。完全に同点。

- `architecture_mermaid.md` プロンプトが Mermaid + ノード表 + エッジ表で構造を完全に保持。
- 視覚的な情報（色分け・矢印の種類）まで text に落とし込めている。
- **結論**: 構成図は specialized プロンプト + Mermaid 変換で text 経路で代替可能。

## 5. Gemini Flash と Flash Lite の品質差は小さいが、一貫して Flash が上回る

Phase 1 の 4 extraction ケース × 2 プロンプト = 8 データポイントで、Flash が Lite を上回ったのは 5/8、同点が 2/8、Lite が上回ったのは 1/8。

- 平均差: Flash 95.1% vs Lite 93.1% (+2.0pp)
- Lite の最大弱点: tc03 の generic (88.5% vs Flash の 94.2%) — 複雑なアーキ図の汎用記述で差が出た。
- **結論**: コスト制約がなければ Flash を使うべき。ただし差は小さいので、レート制限やコストの都合で Lite を使っても大きな品質低下はない。

## 6. LLM の自己採点・交差採点は信頼できない

judgment テストの採点で以下を試したが、いずれも全ケースで 5/5/5/5 を返した:
- **自己採点** (GPT-5.4 が自分の回答を 4 軸で 1-5 評価): 全部 5/5/5/5
- **交差採点** (Gemini が GPT の回答を 4 軸で 1-5 評価): 全部 5/5/5/5

- 原因: LLM は「この回答は良いか？」と聞かれると、回答の内容に関わらず高スコアをつける傾向がある。
- 対策: `reasoning_points` を `present/partial/missing` で fact check する方式に切り替えた。こちらは差別化できるスコアが得られた (tc03_judge で A1=88.9% vs A2=77.8%)。
- **結論**: LLM に品質スコア (1-5) をつけさせるアプローチは、少なくとも現時点のプロンプトでは機能しない。fact check 方式 (present/partial/missing) の方が信頼性が高い。

## 7. Case 1 のプロンプト設計がスコアを大きく左右する (v1 からの再確認)

v1 で発見した「推測禁止の制約が強すぎると LLM が回答を拒否する」問題は v2 でも構造的に重要。

- v2 では最初から「判断には一般的なドメイン知識を自由に使って構いません」と許可しているため、この問題は再発していない。
- **結論**: text 経由で判断を求めるプロンプトでは、「画像の事実は記述を根拠に、判断には一般知識を使ってよい」を明示するのがベストプラクティス。

## まとめ: いつ image が必要で、いつ text で十分か

| 画像の内容 | text 経由 | image 直接 | 推奨 |
|---|---|---|---|
| テキストのみ (仕様書、議事録) | ◎ (100%) | ◎ (100%) | text で十分 |
| 構造化された図 (アーキ図、フロー図) | ○ (90-96%) | ○ (90-92%) | text で十分 (Mermaid 化) |
| 密度の高い混合スライド | ○ (96%) | ○ (92%) | text で十分 (describe 品質次第) |
| 空間的アノテーション (RFP、UI 変更指示) | △ (78%) | ○ (88%) | **image 推奨** |

**全体の結論**: Copilot Chat の画像機能が使えない環境でも、テキスト/構造図/混合スライドは specialized プロンプト + describe スクリプトで**ほぼ同等の品質**で代替できる。ただし、空間的な対応づけが必要な画像（UI 変更要求、地図、注釈付きスクリーンショット等）については、画像を直接渡せる環境の方が明確に優位。

## 既知の限界

- 全結果は 1 回の実行のみ。LLM の非決定性により再実行で数 pp 揺れる可能性がある。
- テスト画像は PIL で生成したクリーンな画像。実際の pptx スライドの解像度・圧縮・背景複雑度での検証は未実施。
- 日本語テキストの OCR 精度は未検証 (tc04 は英語のみ)。
- Copilot Chat での E2E 検証（`copilot-instructions.md` + `run_in_terminal`）は手順書を作成済みだが、スクリーンショット付きの実行記録は未取得。
