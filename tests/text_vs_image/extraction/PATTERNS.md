# Extraction Test Patterns (P1-P8) — v2

現在の [extraction_spec.py](extraction_spec.py) (`CORPUS_VERSION = "v2"`) に含まれる 8 パターンの一覧と内容の snapshot。

- 生成元: `python tests/text_vs_image/extraction/generate_extraction.py`
- 対応する成果物: [extraction_test.pptx](extraction_test.pptx) (8 slides) / `p01_*.png` 〜 `p08_*.png`
- GT: [ground_truth.yaml](ground_truth.yaml)

**総 fact 数**: 273 (P1=30, P2=32, P3=32, P4=35, P5=26, P6=38, P7=41, P8=39)

## v2 設計方針 (v1 → v2 変更点)

v1 (157 facts) では Copilot が PNG 抽出だけで recall 0.89-0.90 を出してしまい、prompt 間の差を判定できない天井効果 (ceiling effect) が発生した。v2 では以下を変更:

1. **密度を約 2 倍に増加** (157 → 273 facts) — テーブル列・行、コード行数、組織図ノード数、コメント数すべてを増量
2. **吹き出しを「答えを書かない」短い略記に変更** — v1 の「KPI カードは 4 枚ではなく 3 枚に減らす」のような自己完結した直接表現は、LLM が画面の他要素を見ずに OCR だけで recall を稼げる。v2 では「5→4」「中央？」「ここ拡大」のような **手書きメモ風の短い表現** を採用し、対象要素は配置・矢印先から推論させる
3. **対象推論ファクトを別個に追加** — 吹き出し本文 verbatim (易) と、その吹き出しが指す対象 (難) を分離した fact として GT に格納

各スライドには共通で以下のメタ情報が含まれる (v1 から継承):
- **Subtitle** (タイトル直下の muted テキスト): ticket 番号 / バージョン / 実施日 / 担当者など
- **Footer** (スライド最下部の muted テキスト): 所属チーム / 機密区分 / 文書バージョンなど

これらは「OCR で拾いやすいが、粗いプロンプトだと省略されがち」な完全性測定要素として機能する。

## 評価設計上の含意

- v2 の recall 期待値は v1 より低くなる (絶対 fact 数増 + vague callout の対象推論項目追加)
- prompt の質的な差 (詳細指示の有無、step-by-step 指示、フォーマット別チューニング) が recall 差として観測されやすくなる
- vague callout 対象推論ファクト (p01 の C1-C6 / p06 の R01-R25) は OCR では拾えず、画像全体の構造理解が要る。これらの recall を別集計することで「OCR 力」と「画像構造理解力」を分離して評価できる
