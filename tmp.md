調査・評価計画（案）

Copilot Chat における画像入力制約下での要件把握精度と業務適用性の評価

1. 背景
	•	当組織では、AI を活用した開発支援施策の一環として、主幹部と POA（Project Owner Assistant）との初期段階での認識合わせを支援するチャット型ツールの活用を検討している。
	•	想定ユースケースは、RFP や構想資料等を入力として、要件、前提条件、未確定事項、確認論点を早期に整理し、後続の feature 定義や開発計画の精度を高めることである。
	•	一方で、実際の入力資料は PowerPoint（PPTX）形式であることが多く、本文テキストだけでなく、表、グラフ、構成図、フロー図、UI モック、画面キャプチャ等を含むケースが多い。
	•	現行の GitHub Copilot Chat（VS Code）利用環境では、Business / Enterprise 利用時の画像入力に関して、組織・enterprise 側の Editor preview features 設定が関わる。また、画像非対応のコンテキストでは画像理解の恩恵は得られず、MCP server を介して間接的に画像情報を補える可能性はあるものの、ネイティブな画像入力の安定した代替手段とまでは言いにくい。
	•	このため、本調査では 「Copilot に画像を直接渡せるか」そのものではなく、PPTX や画像を前処理して構造化し、どこまで要件把握に活用できるかを中心に評価する。

2. 調査目的
	•	画像を直接 LLM に入力できない条件下で、どこまで実用的に要件把握が可能かを明らかにする。
	•	画像を入力できる場合との差分を比較し、画像情報の有用性を定量・定性の両面から整理する。
	•	あわせて、現行のガバナンスの範囲内で採り得る技術的・運用的な実現案を整理する。
	•	最終的には、短期・中期・長期それぞれで取り得る選択肢を示し、意思決定に資する材料とする。

3. 本調査で確認したいこと

本調査は単なる OCR や画像認識の精度比較ではなく、主幹部と POA の初期段階での認識合わせにどの程度役立つかを評価対象とする。
そのため、各ケースについて以下の業務アウトプットを同じ形式で作成し、その品質を比較する。
	•	要件一覧
	•	未確定事項一覧
	•	主幹部に確認すべき質問一覧
	•	feature 分解の初版
	•	論点・リスク一覧

4. 対象・スコープ

4.1 対象
	•	入力資料：RFP、提案資料、構想資料、要件整理資料等の PPTX
	•	利用環境：VS Code + GitHub Copilot Chat
	•	利用者：主幹部、POA、要件整理担当、開発企画担当

4.2 スコープ内
	•	Copilot Chat における画像入力制約の整理
	•	PPTX の原生構造から取得できる情報の整理
	•	画像種別ごとの非 LLM 情報抽出方式の整理
	•	画像あり / 画像なしの比較設計
	•	業務適用性の評価
	•	実行候補案と推奨方針の提示

4.3 スコープ外
	•	本番導入設計
	•	セキュリティ審査の完遂
	•	全社標準ツール選定の最終決裁
	•	長期運用体制の確定

5. 前提・制約
	•	IDE 上での Copilot Chat の画像入力は、現行ポリシー下では前提にしにくい。
	•	GitHub.com 側の画像添付も preview 扱いであり、安定運用を前提とした標準機能とは言い切れない。
	•	MCP は補完手段の候補にはなり得るが、利用可否は組織ポリシーや運用前提に左右されるため、直ちに実運用の前提とは置きにくい。
	•	一方で、PPTX については python-pptx 等により、テキスト、表、グラフ、shape、notes などを構造として扱える。
そのため、資料の一部は「画像認識」の問題ではなく、「PPTX の構造情報をどこまで取り出せるか」という問題として扱える。

6. 調査論点

6.1 論点1

画像入力が使えない前提でも、Copilot に渡す情報を工夫することで、どこまで要件把握の品質を高められるか。

6.2 論点2

PPTX の原生構造から取得できる情報だけで、どこまで業務上使えるアウトプットを作れるか。

6.3 論点3

画像として貼り込まれた要素について、非 LLM の前処理でどこまで補完できるか。

6.4 論点4

画像入力が可能なモデルとの差は、どの種類の資料で特に大きくなるか。

6.5 論点5

短期的に現場で実行可能な案と、中長期的に再評価すべき案は何か。

7. 評価対象の整理

ここでは、PPTX として構造を持っている要素と、画像として貼り込まれている要素を明確に分けて扱う。

7.1 PPTX として構造を持つ要素
	•	スライドのタイトル・本文テキスト
	•	notes
	•	原生の table
	•	原生の chart
	•	text box、shape、connector 等の構造情報

7.2 画像として貼り込まれた要素
	•	文字中心の画像
	•	画像化された表
	•	画像化されたグラフ
	•	フロー図 / 構成図 / アーキテクチャ図
	•	UI モック / 画面キャプチャ
	•	SmartArt / 概念図

8. 想定する技術アプローチ

8.1 スライド上で読めるテキストの抽出
	•	title / body / notes の文字情報を抽出し、最小限のテキストコンテキストを生成する。

8.2 PPTX 原生構造の抽出
	•	原生の table / chart / shape / connector / notes を可能な限り構造化し、テキスト以上の情報として扱う。

8.3 非 LLM による画像前処理
	•	貼り込み画像を抽出し、画像種別ごとに OCR、レイアウト解析、表構造解析、図形・線分解析等を適用する。
	•	出力は Markdown / JSON 等の共通フォーマットに揃える。

8.4 Copilot への入力方式
	•	各ケースで生成された構造化情報を、Copilot Chat が扱いやすい形に整え、比較可能な条件で投入する。
	•	MCP は必要に応じて補完案の一つとして扱うが、本調査では「標準的な安定解」とは位置づけない。

9. 比較ケース

9.1 Case A：スライド上で読めるテキストのみ
	•	title / body / notes の純粋な文字情報のみを使用
	•	table / chart / shape の構造情報は扱わない

9.2 Case B：テキスト + PPTX 原生構造抽出
	•	Case A に加え、原生の table / chart / shape / connector / notes 情報を利用

9.3 Case C：Case B + 非 LLM 画像抽出結果
	•	Case B に加え、画像として貼り込まれた要素に対する OCR、レイアウト解析、表構造解析等の結果を利用

9.4 Case D：画像入力可能なモデルによる理解
	•	比較用ベンチマーク
	•	本番採用候補とは限らない
	•	「画像を直接扱える場合、どこまで差が残るか」を確認するための比較対象

10. 画像種別ごとの確認観点

10.1 原生の表
	•	項目名、値、行列関係をどこまで正確に復元できるか
	•	要件表として再利用可能か

10.2 画像化された表
	•	OCR の後にセル構造まで戻せるか
	•	結合セル、見出し階層、注記がどの程度失われるか

10.3 原生のグラフ
	•	系列、軸、凡例、主要値、傾向をどこまで取得できるか
	•	要件判断や優先度議論に使えるか

10.4 画像化されたグラフ
	•	数値の傾向をどこまで把握できるか
	•	業務解釈に必要な情報がどの程度欠落するか

10.5 原生の shape / text box / connector
	•	ノード、関係、順序、責務分担をどこまで構造として取れるか
	•	フローや構成理解の基礎情報として使えるか

10.6 フロー図 / 構成図 / アーキテクチャ図（画像として貼り込まれたもの）
	•	OCR だけでは不足する関係性がどれだけあるか
	•	誤接続や順序の取り違えがどの程度起こり得るか

10.7 UI モック / 画面キャプチャ
	•	画面部品の列挙は可能か
	•	業務意図、画面遷移、操作前提の把握まで到達できるか

10.8 SmartArt / 概念図
	•	非 LLM 手法でどこまで概念関係を復元できるか
	•	画像入力可能なモデルとの差が特に出やすい領域か

11. 評価指標

11.1 中間評価
	•	要件抽出の網羅性
	•	誤読の発生率
	•	関係性の理解精度
	•	未確定事項の検出率
	•	追加確認質問の妥当性

11.2 業務アウトプット評価
	•	要件一覧の完成度
	•	feature 分解初版の妥当性
	•	論点・リスク一覧の有用性
	•	主幹部と POA の認識合わせに使える水準か
	•	人手レビューの追加負荷がどの程度か

11.3 実行性評価
	•	処理コスト
	•	前処理工数
	•	実装難度
	•	運用の複雑さ
	•	ガバナンス適合性

12. 評価方法
	•	同一資料を用いて Case A～D を実行する。
	•	各ケースで以下の同一成果物を生成する。
	•	要件一覧
	•	未確定事項一覧
	•	主幹部に確認すべき質問
	•	feature 分解初版
	•	論点・リスク一覧
	•	あらかじめ期待値となる基準セットを用意し、差分比較を行う。
	•	必要に応じて有識者レビューを行い、「実際に使えるか」という観点も評価に加える。
	•	比較では特に以下の差分を見る。
	•	A→B：PPTX 原生構造の価値
	•	B→C：非 LLM 前処理の追加価値
	•	C→D：画像を直接扱えるモデルの残る優位性

13. 仮説
	•	テキスト中心の資料であれば、Case A でも一定水準の整理は可能である。
	•	表、グラフ、shape が PPTX の原生オブジェクトとして保持されている場合、Case B の改善幅は大きい。
	•	画像として貼り込まれた要素に対する前処理を加えることで Case C はさらに改善するが、フロー図、構成図、UI モック、概念図では Case D との差が残る可能性が高い。
	•	したがって、画像の価値は単なる補助情報ではなく、関係性の理解、意図の把握、確認質問の質、feature 分解の初速に表れる。

14. 意思決定オプション / 推奨方針

Option 1：短期案

PPTX 原生解析 + 非 LLM 前処理 + Copilot Chat
	•	現行ガバナンス下で最も現実的
	•	まずはこの方式を PoC の主軸とする
	•	期待値は「画像入力の完全代替」ではなく、「実務上十分に使える前処理型支援」

Option 2：中期案

ガバナンス条件を満たしたうえで、MCP 経由の補完を評価
	•	一定の補完可能性はあるが、前提条件が多い
	•	直ちに採用前提とはせず、制度・ポリシー・運用条件を確認したうえで再評価する

Option 3：長期案

利用者に合わせ、IDE 以外の製品形態も再評価
	•	ユーザー主体が開発者ではなく主幹部 / POA である場合、VS Code ベースが最適とは限らない
	•	将来的には IDE 前提から離れた UI / ワークフローも比較対象に含める

推奨
	•	当面は Option 1 を中心に PoC を進める
	•	その結果を踏まえ、必要に応じて Option 2 を評価する
	•	利用者像が固まった段階で Option 3 を再検討する
